# database_homework1
Тема: Система аудиту кібербезпеки мережі
Компанія вирішила створити систему аудиту щоб у разі інциденту спеціалістам було легше відстежити на якому моменті. Ми відстежуємо які користувачі підключилися до системи, які пристрої вони використовували та чи були спраби зламати систему.
Логічна структура таблиць  та тасків:
task 1: Проєктування та створення логічної структури
На цьому етапі було розроблено та створено структуру бази даних, яка складається з 5 взаємопов'язаних таблиць. Усі зв'язки реалізовано через первинні (`primary key`) та зовнішні (`references`) ключі:

1. users: Зберігає дані про працівників.Містить такі колонки:
user_id (INT, PK) — унікальний ідентифікатор
user_name (VARCHAR) — ім'я або логін
role (VARCHAR) — посада через який ми даємо так би мовити рівень доступу в системі
2. divses : Містить інформацію про технічні засоби. Містить такі колонки:
divase_id (INT, PK) — унікальний ідентифікатор пристрою
divase_name (VARCHAR) — назва пристрою
ip_adress (VARCHAR) — IP-адреса пристрою в мережі
3. ecurity_events: Довідник можливих інцидентів та дій. Містить такі колонки:
event_id (INT, PK) — унікальний код події.
event_type (VARCHAR) — тип події
severity_level (INT) — базовий рівень загрози від 1 до 10.
4. logs: Центральна таблиця, яка фіксує хронологію подій.Містить такі колонки:
log_id (INT, PK) — унікальний лог.
user_id (INT, FK) — хто виконав дію
device_id (INT, FK) — з якого пристрою.
event_id (INT, FK) — яка саме подія відбулася.
log_date (TIMESTAMP) — точний час фіксації.
5. audit_results: Фінальний аналіз інцидентів кібербезпеки.Містить такі колонки:
audit_id (INT, PK) — унікальний аналітичний висновок.
log_id (INT, FK) — посилання на конкретний рядок у логах.
risk_score (INT) — реальна динамічна оцінка ризику від 1 до 10.
recommendation (TEXT) — інструкція для офіцера безпеки щодо ліквідації загрози.


task 2: Наповнення таблиць тестовими даними
Для демонстрації працездатності системи таблиці були заповнені реалістичними текстовими та числовими даними (по 5 рядків для довідників та відповідні записи для логів й аудиту). 

task 3: Головний аналітичний SELECT-запит 
Написано головний SQL-запит, який збирає докупи дані з усіх 5 таблиць для створення звіту про безпеку.
Параметри запиту:
Зв'язування -- Фільтрація -- Сортування

task 4: Бонусне завдання (Використання CTE)
Запит створює віртуальну тимчасову таблицю `high_risk`
